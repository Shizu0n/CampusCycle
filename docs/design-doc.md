## Problem Statement

Desafio técnico de processo seletivo para estágio full-stack (Laboratório Vortex, UNIFOR), prazo de 15 dias. Entregar um marketplace de desapego universitário: API RESTful (CRUD de anúncios com filtros, JSON, persistência) + frontend PWA responsivo (manifest.json válido, Service Worker instalável, landing page rica no desktop, experiência de app no mobile com formulário de anúncio e listagem "meus anúncios").

O objetivo real do candidato não é construir o produto perfeito: é **maximizar a nota nos 4 eixos da rubrica** — (1) qualidade da entrega (Git + README), (2) domínio técnico e autoria provados num vídeo de 6 minutos cronometrados, (3) requisitos obrigatórios, (4) uso inteligente e documentado de IA (Diário de Bordo obrigatório; omissão penaliza severamente). Dois dos quatro eixos são artefatos de comunicação, não código.

## What Makes This Cool

1. **Fila offline de anúncios (o momento-uau):** o estudante está numa zona sem sinal do campus, preenche o formulário de anúncio, o app guarda em IndexedDB com badge "aguardando sincronização"; ao reconectar, a fila sincroniza e o badge vira "publicado". No vídeo: ligar modo avião na câmera, anunciar, reconectar, ver aparecer. Quase nenhum candidato terá isso, e prova domínio real de arquitetura offline — exatamente o eixo de autoria.
2. **Contador de impacto real na landing:** "X kg desviados do lixo, R$ Y economizados pela comunidade" calculados dos anúncios reais (peso estimado por categoria, com piso simulado para os números não parecerem vazios com poucos anúncios — o edital pede "estatísticas simuladas", então o piso é fiel ao enunciado). Amarra o tema "economia circular" que dá nome ao desafio.
3. **Sistema vivo em produção:** links funcionais de API e frontend no README — o "fortíssimo diferencial" nas palavras do próprio edital.

## Constraints

- **Prazo:** 15 dias, com disponibilidade regular do candidato (premissa 5).
- **Autoria:** todo código precisa ser explicável pelo candidato num vídeo de 6 min; complexidade que não se domina é passivo, não ativo.
- **Free tiers (verificado em jul/2026):** Fly.io sem free tier (cartão obrigatório); Railway só trial de 30 dias; Render mantém free tier real, mas o serviço hiberna e leva ~1 min para acordar (cold start); Postgres free do Render expira em 30 dias → usar Neon (ou Supabase) para o banco.
- **PWA só é testável em build de produção** (`vite build` + `vite preview`), não no dev server.
- **A fila offline usa listener de `online` + retry no load como implementação primária** (funciona em qualquer navegador e é totalmente explicável no vídeo). Background Sync API é Chromium-only e duplicaria essa lógica com risco de post duplo — fica como stretch opcional. A demo deve ser gravada em Chrome (Android ou DevTools device mode).
- **Vídeo:** máximo estrito de 6 minutos, estrutura cronometrada fixa (1 min pitch / 2 min demo / 2 min código / 1 min IA).
- Ambiente de desenvolvimento: Windows 11.

## Premises

1. O objetivo é maximizar a nota nos 4 eixos da rubrica — decisões de escopo seguem a rubrica, não o produto ideal.
2. Stack: Node.js + Express + TypeScript (backend); React + Vite + TypeScript + vite-plugin-pwa (frontend); monorepo único e público no GitHub.
3. Deploy real desde a primeira semana: Render (API) + Vercel (frontend) + Neon (Postgres), com mitigação de cold start (ping periódico via UptimeRobot/cron-job.org + aquecer a API antes de gravar).
4. Diário de Bordo da IA alimentado desde o dia 1 (prompts reais colados na hora) e commits incrementais diários como evidência de autoria — nunca reconstruído na véspera.
5. O candidato tem disponibilidade regular nos 15 dias e grava o vídeo seguindo o roteiro cronometrado.

## Cross-Model Perspective

Segunda opinião via subagente Claude independente (Codex indisponível — limite de uso). Leitura fria a partir de resumo estruturado, sem acesso à conversa:

- **Versão mais legal não considerada:** fila offline de anúncios com Background Sync + IndexedDB e badge de sincronização visível — "o PWA deixa de ser 'um site com manifest' e vira um app de verdade"; cena de modo avião como o melhor momento possível do vídeo. Custo estimado: ~1 dia sobre o plano. Ideia barata no mesmo espírito: contador de impacto real na landing. (Ambas adotadas.)
- **O que revela o candidato:** o entusiasmo está em "colocar um sistema real no ar e ele não te envergonhar", não no CRUD — instinto certo para esta rubrica, com alerta: não deixar a empolgação com infra roubar tempo do polimento mobile. (Adotado como restrição de cronograma.)
- **Projeto de referência:** RealWorld/"Conduit" (gothinkster/realworld) — Express+TS e React+TS com JWT, CRUD de posts por usuário, filtros por tag e validação: ~50% da arquitetura como referência. Aviso enfático: **não forkar** — estrutura reconhecível de terceiros no vídeo de autoria é pior que código próprio mais simples. Ler, absorver padrões, digitar o próprio código. (Adotado.)
- **O que construir primeiro:** walking skeleton deployado ponta a ponta antes de qualquer polimento — deploy e instalabilidade PWA são os dois modos de falha que não se consertam na véspera; risco de integração é incompressível, polimento é compressível. (Adotado como espinha dorsal do cronograma.)
- Nenhuma premissa foi desafiada; a premissa 3 (deploy semana 1) foi endossada.

## Approaches Considered

### Approach A: Núcleo Blindado (mínimo viável)
Todos os obrigatórios impecáveis + deploy + validação. Sem JWT nem fila offline; identificação por ID de usuário em localStorage; SW com precache básico. Esforço M (~8-10 dias), risco baixo. Prós: folga máxima para README/diário/vídeo; menos código para defender. Contras: destaque vem só do deploy e acabamento — candidato fica na média dos organizados.

### Approach B: Pacote de Bônus Completo (arquitetura ideal)
A + JWT, Postgres real (Neon) com Prisma, TypeScript ponta a ponta, cache offline NetworkFirst para leitura, UI polida com estados de loading e transições. Esforço L (~12-13 dias), risco médio. Prós: marca todos os bônus do edital. Contras: menos folga; mais superfície para a banca sondar.

### Approach C: B + Momento-Uau (escolhida)
B + fila offline de anúncios (IndexedDB + listener `online`; Background Sync apenas como stretch), contador de impacto real, vídeo roteirizado em torno da cena do modo avião. Construída em fases: esqueleto deployado até o dia 3, B completo até o dia 10, uau nos dias 10-12 — se atrasar, degrada para B sem retrabalho. Esforço L/XL (~13-14 dias), risco médio com fallback embutido.

## Recommended Approach

**Approach C**, porque o objetivo declarado é "ir além dos requisitos mínimos para ganhar destaque", o candidato já construiu projetos com Node e React, e a estrutura em fases faz C degradar graciosamente para B. Escolhida pelo usuário (D6).

### Modelo de dados (Prisma / Neon Postgres)

```prisma
model User {
  id        String    @id          // no estágio 1, é o próprio UUID anônimo do localStorage (linha stub criada por upsert no primeiro POST — sem isso a FK de Listing quebraria); no estágio 2, o registro ganha name/email/password reais no cadastro
  name      String    @default("Anônimo")
  email     String?   @unique      // null até o cadastro (estágio 2)
  password  String?   // bcrypt hash; null até o cadastro
  listings  Listing[]
  createdAt DateTime  @default(now())
}

model Listing {
  id          String   @id     // UUID gerado no CLIENTE — dá idempotência à fila offline (reenvio do mesmo id = no-op)
  title       String
  description String
  category    String   // Livros | Engenharia | Computação | Eletrônicos | Vestuário | Móveis | Outros
  price       Int?     // em centavos; null = doação (evita o Decimal do Prisma, que serializa como string no JSON)
  imageUrl    String?  // opcional; vazio = imagem placeholder da categoria
  status      String   @default("available") // available | sold | donated — alterável via PATCH
  user        User     @relation(fields: [userId], references: [id])
  userId      String
  createdAt   DateTime @default(now())
}
```

### Endpoints

| Método | Rota | Auth | Descrição |
|---|---|---|---|
| POST | /api/auth/register | — | cria usuário, retorna JWT |
| POST | /api/auth/login | — | retorna JWT |
| GET | /api/listings | — | lista com `?category=&q=&donation=true&page=` |
| GET | /api/listings/:id | — | detalhe (consumido pela página ListingDetail mínima — clique no card) |
| POST | /api/listings | identidade* | cria anúncio (validação zod; id UUID vem do cliente — reenvio do mesmo id é idempotente) |
| PATCH | /api/listings/:id | identidade* (dono) | atualiza título/descrição/preço/status (ex: marcar como vendido) — completa o U do CRUD |
| GET | /api/listings/mine | identidade* | meus anúncios |
| DELETE | /api/listings/:id | identidade* (dono) | remove anúncio |
| GET | /api/stats | — | totais + impacto (kg/R$ estimados por categoria) |
| GET | /api/health | — | usado pelo ping anti-cold-start |

\* **Contrato de identidade em dois estágios** (o requisito obrigatório "meus anúncios" nunca depende do bônus de auth). As rotas são idênticas nos dois estágios — só o middleware de identidade muda:
- **Middleware de identidade:** resolve `req.userId` a partir do header `X-User-Id` (estágio 1) ou do JWT (estágio 2). Todas as rotas marcadas com identidade* usam `req.userId` — inclusive `/api/listings/mine`, que existe desde o dia 4 com a mesma forma nos dois estágios. (Nota de implementação: registrar `/mine` ANTES de `/:id` no Express, senão `:id` engole a rota.)
- **Estágio 1 (dias 4-5, e fallback permanente se o auth for cortado no dia 8):** o cliente gera um `userId` UUID anônimo e guarda em localStorage, enviado via `X-User-Id`. No primeiro POST, o servidor faz **upsert de uma linha stub em `User`** com esse UUID como id (satisfaz a FK de `Listing`; ver comentário no schema). O servidor confere `userId === listing.userId` para PATCH/DELETE. Confiança fraca — documentada no README como escopo de demo, defendida no vídeo como decisão consciente.
- **Estágio 2 (dias 6-7, bônus):** o cadastro preenche name/email/password na linha stub existente (o UUID anônimo VIRA o id da conta — migração trivial por construção) e o middleware passa a exigir JWT nas rotas de escrita. Nenhuma rota nem chamada do cliente muda de forma.

Erros sempre em envelope JSON consistente: `{ "error": { "code", "message", "details?" } }`. Validação com zod em todas as rotas de escrita (campos obrigatórios, tipos, limites) — cobre o bônus "tratamento robusto de erros".

### Estratégia PWA (vite-plugin-pwa, modo injectManifest)

- **Precache** do app shell (HTML/CSS/JS hasheados) — instala e abre offline.
- **NetworkFirst** para `GET /api/listings*` — feed atualizado online, último feed visível offline (bônus de visualização offline).
- **CacheFirst** para imagens externas com expiração.
- **NetworkOnly** para auth.
- **Fila offline (uau):** `POST /api/listings` falhou por rede → salva em IndexedDB (lib `idb`) com o UUID gerado no cliente, UI mostra badge "aguardando sincronização". Sync primário: listener de `online` + retry no load do app — simples, cross-browser, e o UUID do cliente garante que retry duplo não cria anúncio duplicado. Background Sync API é **stretch opcional** (só se os dias 10-11 sobrarem) — duplicaria a lógica com alto custo e zero ganho de demo. JWT com expiração de 7 dias para que um anúncio enfileirado horas antes ainda sincronize (escolha deliberada, documentada).
- **Atualização:** `registerType: 'prompt'` + `useRegisterSW` (virtual:pwa-register/react) para toast de nova versão — `prompt` é o modo em que o evento `needRefresh` realmente dispara (com `autoUpdate` o toast seria código morto).
- manifest.json completo: name, short_name, theme_color, background_color, display standalone, ícones 192/512 + maskable, screenshots (capturados da produção no dia 13).
- **Modo do plugin: `injectManifest` (SW escrito à mão em `sw.ts`) — escolha deliberada pela rubrica:** `generateSW` cobriria as mesmas estratégias via config, mas config não é código explicável; um `sw.ts` autoral de ~60 linhas é exatamente o material que o vídeo de autoria pede no trecho "lógica do Service Worker". Se o SW manual travar o cronograma no dia 8, degradar para `generateSW` + `runtimeCaching` sem perda funcional.
- Testar sempre com `vite build && vite preview` + DevTools > Application > Manifest (checks de instalabilidade — a auditoria "PWA" do Lighthouse foi removida em 2024 e não existe mais) + instalação real no dispositivo de demo.

### Deploy e CI/CD

- **API:** Render free (auto-deploy on push da pasta `server/`). Cold start ~1 min mitigado por ping a cada 10 min no `/api/health` (UptimeRobot ou cron-job.org) + aquecimento manual antes de gravar o vídeo.
- **Banco:** Neon Postgres free (não expira, ao contrário do Postgres free do Render). **Atenção ao pooling:** usar a connection string *pooled* (pgbouncer) do Neon no `DATABASE_URL` do Prisma; a string direta só para migrations. O Neon também hiberna no free tier — os cold starts de Render e Neon se somam, então o aquecimento pré-gravação deve acertar um endpoint que toca o banco (ex: `/api/stats`), não só o `/api/health`.
- **Frontend:** Vercel (auto-deploy on push da pasta `web/`), variável `VITE_API_URL`.
- **CORS** configurado para o domínio do Vercel.

### Micro-decisões travadas (para não parar no meio da implementação)

- **Imagem do anúncio:** campo de URL *opcional* no formulário; vazio → imagem placeholder da categoria (7 imagens locais no bundle). Colar URL num celular é estranho — o placeholder mantém a demo fluida e a fila offline só carrega strings (sem blob/upload). Upload real fica explicitamente fora de escopo.
- **Tabela de impacto (kg por categoria):** Livros 0.8 | Engenharia 1.5 | Computação 1.2 | Eletrônicos 0.5 | Vestuário 0.4 | Móveis 8.0 | Outros 1.0. R$ economizado = soma dos preços dos itens vendidos + valor médio da categoria para doações.
- **Seed:** 20-30 anúncios cobrindo todas as categorias (imagens Unsplash), volume calibrado para o contador de impacto exibir números apresentáveis; piso simulado somado ao valor real (fiel ao "estatísticas simuladas" do edital).
- **JWT:** armazenado em localStorage; tradeoff de XSS reconhecido e defendido no vídeo ("para produção real usaria cookie httpOnly — escolha consciente de escopo de demo"). Expiração 7 dias.
- **Paginação:** `?page=` com `take/skip`, tamanho fixo 12; se o feed com "carregar mais" ameaçar o cronograma, cortar sem dó (não é requisito do edital).

### Fila offline (o uau) — máquina de estados obrigatória

1. **Estados por item (Issue 1):** `pending | synced | failed`, com `errorReason` e `retryCount` persistidos no IndexedDB.
   - Falha de REDE (fetch rejeita) → permanece `pending`, retry no evento `online` e no load.
   - `4xx` de validação → `failed`, badge "falhou — toque para revisar" (reabre o formulário preenchido).
   - `401` (JWT expirado) → permanece `pending`, UI pede re-login; sincroniza após.
   - `429` (rate limit) → permanece `pending`, retry com backoff.
   - `5xx`/limite de 5 retries → `failed` com motivo. **Nenhum estado é silencioso.**

```
                      rede falhou
  [form submit] ────────────────────▶ PENDING ──── online/load retry ────┐
       │                                │  ▲                            │
       │ 2xx                        401 │  │ re-login                   ▼
       ▼                                │  │                     POST /api/listings
    SYNCED ◀── 2xx ou P2002 ────────────┼──┴──────────────────────────  │
                                        │                               │
                                        ▼ 4xx / 5º retry / 5xx          │
                                     FAILED ── "toque p/ revisar" ──▶ volta ao form
```
